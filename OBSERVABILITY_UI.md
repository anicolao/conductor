# Conductor Observability UI: Design & Implementation Plan

## Overview

The Conductor Observability UI is a web-based dashboard designed to provide "Transparent Orchestration" for the `LLM-Orchestration` ecosystem. It enables developers and project managers to follow Conductor's multi-persona workflows in real-time, analyze historical runs, and understand the "blow-by-blow" logic of AI agents.

## Core Principles

1.  **Zero-Backend Architecture**: Hosted on GitHub Pages as a SvelteKit SPA, communicating directly with the GitHub API.
2.  **Authentication as a Utility**: Users provide their own Personal Access Token (PAT), which is stored locally in the browser.
3.  **Source of Truth: GitHub**: No external database. Status is derived from Project V2 boards, Issue labels/comments, and GitHub Actions logs.
4.  **Cross-Repo Visibility**: A unified view of all repositories managed by Conductor within the organization.

---

## Technical Architecture

### Tech Stack
-   **Framework**: [SvelteKit](https://kit.svelte.dev/) (Static Site Generation / SPA mode).
-   **Styling**: Vanilla CSS or a lightweight framework like [Skeleton](https://www.skeleton.dev/) (Tailwind-based) for rapid UI development.
-   **GitHub Client**: [@octokit/core](https://github.com/octokit/core.js) and GraphQL for efficient Project V2 data fetching.
-   **Deployment**: GitHub Actions deploying to GitHub Pages.

### Authentication
-   **Method**: The UI will prompt the user for a GitHub Personal Access Token (PAT).
-   **Permissions Required**: `repo`, `workflow`, `project`, `read:org`.
-   **Storage**: Stored in `localStorage`. Never transmitted to any server other than `api.github.com`.

### Data Flow
1.  **Discovery**: Use GraphQL to fetch items in the `AI Orchestration` Project V2 board.
2.  **Linking**: Map project items to their source issues/PRs across the organization.
3.  **Action Retrieval**: For a selected issue, find associated GitHub Actions runs in the `LLM-Orchestration/conductor` repository.
4.  **Log Extraction**: Fetch job logs for a specific run ID and parse them to extract persona activity.

## UI Views

### 1. Organization Dashboard
-   **Project Overview**: Visual representation of the `AI Orchestration` board.
-   **Active Items**: Highlight issues in the `In Progress` column.
-   **Persona Badges**: Show which persona is currently assigned to each active item.
-   **Recent Activity**: A feed of the latest workflow runs across the organization.

### 2. Workflow Detail View (Real-time & Historical)
-   **Status Header**: Current status (Queued, In Progress, Completed, Failed).
-   **The "Blow-by-Blow" Log**:
    -   A formatted terminal-like output.
    -   Filters to show/hide raw logs vs. persona-specific thoughts/actions.
    -   Syntax highlighting for code blocks generated by agents.
-   **Persona Timeline**: A visual horizontal timeline showing transitions (e.g., `Conductor` -> `Coder` -> `Conductor`).
-   **Artifacts**: Links to PRs created or modified during the run.

### 3. Repository Browser
-   List of all repositories in the `LLM-Orchestration` org.
-   Quick access to "Conductor-enabled" repos.

## Implementation Details

### Real-time Visibility
GitHub Actions logs do not support native WebSockets for streaming to 3rd party apps. We will implement:
-   **Polling Strategy**: While a run is `in_progress`, the UI will poll the `actions/runs/{run_id}/jobs` endpoint every 5-10 seconds.
-   **Log Tailing**: Fetch the latest log chunks for the active job. Since full logs can be large, we will use the `Range` header or simply fetch the whole log if it's within reasonable limits for a single persona run.

### Log Parsing Logic
Conductor agents output structured markers in their logs. The UI will parse these to provide a rich experience:
-   `> [persona] Thinking...`: Rendered as a collapsible thought block.
-   `> [persona] Action: [command]`: Rendered as a command execution block.
-   `> [persona] Result`: Success/Failure indicators.

### Project V2 Integration
Using the GitHub GraphQL API, we will fetch the custom `Persona` field to show real-time handoff state even before a workflow run starts.

```graphql
query($org: String!, $number: Int!) {
  organization(login: $org) {
    projectV2(number: $number) {
      items(first: 20) {
        nodes {
          id
          fieldValues(first: 10) {
            nodes {
              ... on ProjectV2ItemFieldSingleSelectValue {
                name
                field { ... on ProjectV2Field { name } }
              }
            }
          }
          content {
            ... on Issue { title, number, repository { name } }
          }
        }
      }
    }
  }
}
```

## Implementation Plan

### Phase 1: Foundation (Week 1)
1.  Initialize SvelteKit project with Static Adapter.
2.  Set up GitHub Pages deployment workflow.
3.  Implement PAT input and persistence logic.
4.  Basic layout with "Settings" and "Dashboard" routes.

### Phase 2: Data Integration (Week 1-2)
1.  Implement GraphQL service for Project V2 board.
2.  Create "Project Board" view component.
3.  Implement "Workflow List" for a specific issue.

### Phase 3: The Observability Engine (Week 2-3)
1.  Implement log fetching and polling logic.
2.  Build the "Persona Log" parser and renderer.
3.  Add "Real-time" indicator and auto-refresh.

### Phase 4: Polish & Historical Analysis (Week 3)
1.  Timeline visualization for handoffs.
2.  Filterable history for past workflows.
3.  Performance optimizations for large log files.

---

## Success Criteria
-   A user can paste their PAT and immediately see all `In Progress` tasks in the `LLM-Orchestration` org.
-   Clicking a task shows the live log of the running GitHub Action.
-   Persona transitions are clearly visible without reading raw YAML/logs.
-   Historical runs are accessible via issue number.
